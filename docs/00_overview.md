# 00_overview — TryOnLab

## 1) Что это
**TryOnLab** — отдельный тестовый Telegram-бот для быстрой проверки функции «виртуальная примерка»:

- вход: **фото товара** + **фото человека**
- выход: **1 try-on изображение** (и по желанию **короткое видео** на основе результата)

Проект **не зависит** от SIS/SizeBot на первом шаге, но **закладывает интерфейсы** для последующей интеграции (подтягивать мерки/параметры одежды по API).

Ключевой принцип: **провайдеры генерации сменяемые** (Grok / OpenAI / любые другие), выбираются пользователем прямо в боте.

---

## 2) Цели MVP (V1)
### Пользовательский результат
1) Получить **реалистичное try-on изображение** (сохранить лицо/позу/фон человека, заменить одежду).
2) Иметь **кнопку “Регенерировать”** (ретрай) без повторной загрузки фото.
3) При хорошем результате — **сгенерировать видео** по 5 фиксированным пресетам (кнопки `1..5`).

### Инженерные цели
- Быстрое прототипирование: минимальный, но «продакшн-аккуратный» каркас (API+бот+очередь).
- Плагинная архитектура: добавление провайдера = новый файл в `app/providers/` + регистрация.
- Детерминированная управляемость: **job-ориентированная** модель (создал задачу → статус → результат).

---

## 3) Что НЕ делаем в V1
- Интеграция с SIS (каталог, карточки, медиаконтент).
- Интеграция с SizeBot (автовытягивание мерок / size verdict).
- Авто-вытаскивание “лучшего кадра” из видео товара.
- Мультиракурс try-on (front/side/back) как обязательный стандарт.
- Платёжка / поинты / подписки.
- Персонализация «по гардеробу пользователя», лонг-стори и т.д.

---

## 4) Основные сценарии (UX)
### Wizard (все через inline-кнопки, минимум текста)
1) **Загрузить фото товара**
2) **Загрузить фото человека**
3) **Выбор посадки (fit)**: `slim / regular / oversize`
4) **Опционально**: рост + мерки (одно сообщение)
5) **Генерация изображения** → показать результат + кнопки:
   - `🔁 Регенерировать`
   - `🎬 Видео (1..5)` (доступно после удачного изображения)
   - `⚙️ Провайдер: Grok | OpenAI`
   - `🧼 Сбросить`

> В V1 допускается, что мерки могут не давать заметного прироста. Мы проверяем это на практике, но собираем UX уже сейчас.

---

## 5) Архитектура (как в SocialBridge)
Проект состоит из двух активных компонентов:

1) **Telegram Bot (aiogram)**  
   Отвечает за UX: wizard, загрузки, переключение провайдера, кнопки, отображение статусов.

2) **API (FastAPI)**  
   Отвечает за: задачи (jobs), вызов провайдера, хранение результатов, контроль ошибок/ретраев, выдачу статуса.

Дополнительно:
- **Postgres** — долговременное хранение: пользователи, задачи, результаты, настройки.
- **Redis** — быстрые статусы/локи/TTL, rate-limit, временные ключи.
- **Storage** (S3/MinIO или локально в V1) — временное хранение входных/выходных медиа.

### Диаграмма потока (логическая)
Telegram Bot
- POST /jobs (create)
- GET /jobs/{id} (poll)
API
- provider.generate_image(...)
- provider.generate_video(...)
- save result_url + metadata
- update job status

## 6) Провайдеры генерации (ключевой дизайн)
В `app/providers/` находится набор провайдеров.

### Контракт провайдера
Любой провайдер обязан реализовать:
- `generate_image(inputs, prompt_pack) -> ProviderResult`
- `generate_video(image_result, video_prompt) -> ProviderResult`
- `capabilities()` (флаги: поддержка video, async_video, image_edit и т.д.)

### Выбор провайдера
- Пер-пользовательский: хранится в БД (`user.provider`).
- Ограничение через allowlist (`AI_PROVIDER_ALLOWLIST`) для быстрого отключения “упавшего” провайдера.
- Default (`AI_PROVIDER_DEFAULT`).

---

## 7) Промпты и пресеты
### Промпт для изображения (Try-on)
Цель: **не менять лицо/волосы/фон**, заменить одежду на товар.

- Базовый системный промпт зашит в `app/services/prompts.py`
- Переменные из wizard: `fit_pref`, (опц.) `height`, `measurements`

### Пресеты видео (кнопки 1..5)
Каждая кнопка = фиксированный “сценарий” движения:
1) 2 шага вперёд + поворот 45°
2) Поворот 180° с паузой
3) Лёгкая подиумная проходка + поза
4) Движение рук (актуально для верхней одежды)
5) “Ткань/драпировка” (минимальная динамика)

> В V1 не делаем сложную хореографию. Приоритет: стабильность лица и анатомии.

## 8) API (минимальный набор)
### Endpoints
- `POST /jobs`
  - body: ссылки/байты изображений + параметры (fit, мерки) + provider
  - returns: `job_id`
- `GET /jobs/{job_id}`
  - returns: `status`, `progress`, `result_image_url`, `result_video_url` (если есть), `error`
- `POST /jobs/{job_id}/retry`
  - перегенерация изображения с новым seed/вариантом
- `POST /jobs/{job_id}/video`
  - запуск видео по preset 1..5

---

## 9) Данные и хранение (privacy-first)
### Входные фото
- Хранение **временно**: TTL (например 72 часа) → авто-очистка.
- Храним минимум: ссылки/ключи на storage + размеры файлов + хэши.

### Результаты
- Изображения/видео — тоже с TTL, если это тестовый режим.
- Метаданные результата (время генерации, провайдер, модель, успех/ошибка) — остаются для анализа качества.

### Требования безопасности
- Никаких ключей в коде, только ENV.
- Логи без “сырых” фото/ссылок, максимум job_id.
- Rate-limit по пользователю/дню.

---

## 10) Метрики качества (что считаем)
- % успешных генераций (image/video)
- Среднее время генерации (p50/p95)
- Среднее число ретраев до “принятого” результата
- Ошибки по типам (auth/rate-limit/bad-request/timeout)
- Влияние мерок: A/B (с мерками vs без) на субъективную оценку результата

---

## 11) ENV (черновой список)
- `TELEGRAM_BOT_TOKEN`
- `DATABASE_URL`
- `REDIS_URL`
- `STORAGE_URL` (или локальный путь)
- `RETENTION_HOURS=72`
- `AI_PROVIDER_DEFAULT=grok`
- `AI_PROVIDER_ALLOWLIST=grok,openai`
- `XAI_API_KEY`
- `OPENAI_API_KEY`
- `OPENAI_IMAGE_MODEL` (опц.)
- `OPENAI_VIDEO_MODEL=sora-2` (опц.)
- `HTTP_TIMEOUT_SECONDS=60`
- `MAX_RETRIES=2`

## 12) Roadmap (V2+)
### V2 (полезное)
- Интеграция SizeBot (подтягивание мерок + size verdict)
- “Hero frame extraction” из видео товара с выбором кадра в админ-UI

- Поинты/платёжка: лимиты + монетизация видео

### V3 (красиво, но позже)

- Персональные стили/луки, гардероб, рекомендации
- Стабильная “анатомия” и локальные маски/поза-контроль (если потребуется)

---

## 13) Правила для Codex (обязательные)
- Провайдеры только через `ProviderBase` и registry.
- Любая генерация = **job** (никаких “сразу в боте без статуса”).
- Все вызовы провайдеров через `httpx`, с таймаутами и ретраями.
- Никаких хардкодов ключей, URL и моделей. Только ENV/конфиг.
- Изменение структуры проекта — только если приносит измеримую пользу (и описано в docs).
