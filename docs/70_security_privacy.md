# 70_security_privacy — TryOnLab (Security, Privacy, Data Retention)

## 0) Зачем этот документ
TryOnLab работает с самым “приятным” типом данных: **фото человека**.  
Значит, по умолчанию:
- минимизируем хранение
- даём пользователю контроль (удалить/заменить)
- защищаем ключи и ссылки
- логируем аккуратно
- ставим лимиты, чтобы не стать бесплатным рендер-фермером для интернета

---

## 1) Модель угроз (коротко)
### 1.1 Что может пойти не так
- Утечка API ключей (xAI/OpenAI) → деньги улетели.
- Утечка фото пользователя → репутационный и юридический ад.
- Хранение “вечно” → накопление риска.
- Временные ссылки провайдеров попали в логи/чат → кто угодно скачает.
- Спам/абьюз: бот используется как генератор контента для перепродажи.

### 1.2 Приоритеты
1) Ключи и секреты
2) Контроль хранения фото
3) Ограничение доступа к медиа
4) Anti-abuse

---

## 2) Данные: какие именно и зачем
### 2.1 Пользовательские фото
- `user_photos`: фото человека, которое пользователь загрузил для повторного использования
- `job_inputs`: фото товара и ссылка на user photo/временное фото

### 2.2 Метаданные
- `measurements_json`: рост/мерки (опционально)
- `fit_pref`
- `provider`, `model`, `prompt_version`
- `status`, `attempts`, `error_code`

### 2.3 Что не храним
- Никаких паспортных данных, адресов, телефонов (кроме Telegram id, который и так есть).
- Никаких “профилей личности” и биометрии.
- Никаких “сырых” ответов провайдера, если там есть токены/подписи/временные URL.

---

## 3) Политика хранения (retention)
### 3.1 Job media (входы/выходы задач)
- По умолчанию: `RETENTION_HOURS = 72`
- После истечения:
  - job помечаем `expired`
  - **удаляем** input/output ключи из storage
  - метаданные jobs оставляем (для метрик качества)

### 3.2 User photos (реюз)
По умолчанию:
- Храним **до удаления пользователем**
- Пользователь в любой момент может:
  - удалить одно фото
  - удалить все
  - загрузить новое и сделать его active

Опционально (если нужно):
- `retention_days_user_photos` (например 30/90 дней) на уровне settings
- авто-cleanup удаляет старые фото

### 3.3 Результаты в Telegram-чате
- Bot **не удаляет** сообщения с результатами (image/video).
- Это сознательное решение UX: пользователь сам чистит чат, если хочет.
- Но storage-объекты под результатами **всё равно** удаляются по TTL, если это job-scope.

> Важно: если вы шлёте пользователю видео как “файл” (upload), Telegram будет хранить копию на своей стороне. Bot не контролирует срок хранения Telegram.

## 4) Контроль пользователя (обязательные команды/кнопки)
### 4.1 Управление user-фото
В меню “👤 Моё фото”:
- `Выбрать активное`
- `Добавить новое`
- `Удалить выбранное`
- `Удалить все`

Поведение:
- если удалили active — выбираем следующее или требуем загрузку

### 4.2 Сброс сессии
`🧼 Сбросить сессию`:
- очищает только session state (product image, fit, measurements)
- не удаляет user photos

### 4.3 “Удалить мои данные” (рекомендовано)
Кнопка/команда:
- удаляет:
  - user_photos (storage + DB)
  - user_settings
  - jobs метаданные (по выбору) или анонимизирует (tg_user_id hash)
- оставляет минимум технических логов без привязки к личности (если надо)

---

## 5) Хранилище медиа (Storage security)
### 5.1 MinIO/S3
Рекомендации:
- бакет **private**
- доступ только по сервисным ключам внутри сети
- выдача наружу только через:
  - **проксирование через API** (подписанные ссылки с TTL)
  - или presigned URL короткой жизни

### 5.2 Временные URL провайдера
Некоторые провайдеры отдают **временные URL**.
Правило:
- **не хранить** эти URL “как конечный результат”.
- worker должен **скачать файл** и положить в наш storage, а в БД хранить только наш ключ.

Причина:
- провайдерский URL истечёт
- попадёт в лог/чат
- станет источником утечек

---

## 6) Secrets management
### 6.1 Где держим ключи
- только в `.env` на сервере (права 600)
- никогда не коммитим
- никогда не печатаем в логи

### 6.2 Ротация ключей
- смена ключа → обновили `.env` → `docker compose restart api worker bot`
- при подозрении на утечку: немедленно revoke + новые ключи

### 6.3 Обязательные проверки
- при старте сервиса валидируем:
  - если провайдер включён в allowlist, то ключ должен быть установлен, иначе сервис пишет ERROR и отключает провайдер (или не стартует)

---

## 7) Network / Transport
- Все внешние вызовы только по HTTPS.
- Таймауты обязательны (`HTTP_TIMEOUT_SECONDS`).
- Ретраи ограничены (`MAX_RETRIES`) и только для retryable ошибок.
- При возможности на сервере: egress allowlist по доменам провайдеров.

---

## 8) Логи (privacy-safe logging)
### 8.1 Что логируем
- `job_id`, `provider`, `status`, `attempt`
- HTTP статус и длительность запроса
- `error_code` (классификация)

### 8.2 Что НЕ логируем
- ссылки на медиа, если они содержат подписи/токены
- полный текст промпта (можно хранить “prompt hash”)
- сырые JSON ответы провайдера целиком
- мерки пользователя в открытом виде (если логируем — то только факт наличия и количество полей)

### 8.3 Режим debug
Debug режим допустим только локально и должен быть выключен на проде.

---

## 9) Anti-abuse (защита от халявщиков)
### 9.1 Rate limits
- общий лимит jobs/час на пользователя
- отдельный лимит видео (дороже)
- реализуем через Redis counters

### 9.2 Cooldown
- если пользователь жмёт retry слишком часто — cooldown 30–60 сек

### 9.3 Quotas / поинты (V2)
- кредитная система: image дешевле, video дороже
- это не про “жадность”, а про выживание бюджета

---

## 10) Доступ и права
### 10.1 Админ-доступ
Если будет админ-режим:
- админ не должен видеть чужие фото “по списку”
- любые просмотры только по явному job_id, и то лучше без доступа к raw media

### 10.2 Разделение окружений
- dev и prod ключи разные
- dev storage отдельный бакет

---

## 11) Нормы поведения бота (UX как часть приватности)
- перед сохранением user-фото показываем короткое уведомление в панели:
  - “Фото сохранено для повторного использования. Можно удалить в ‘Моё фото’.”
- в настройках даём “Удалить мои данные”
- не просим лишнего: мерки и рост — опционально

---

## 12) Checklist (Definition of Done)
**V1 Security/Privacy готово, если:**
- ✅ входные фото (товар/человек) удаляются из чата (best-effort) после сохранения
- ✅ user-фото сохраняется и может быть удалено (одно/все)
- ✅ job media удаляется по TTL
- ✅ временные URL провайдера не сохраняются как результат (скачиваем в storage)
- ✅ ключи только в env, не в логах, не в репо
- ✅ rate-limit включён
- ✅ есть команда/кнопка “Удалить мои данные” (минимальная реализация)
