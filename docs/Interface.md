TL;DR

Одна панель (одно сообщение) всегда “на экране”, всё управление только из неё.

Нажал кнопку → панель перерисовалась в другой экран (edit_text + edit_markup), старых не плодим.

Где надо текст: нажал → сразу режим await_text, бот кидает ForceReply/инструкцию.

Где фото: нажал → сразу режим await_photo, юзер просто кидает фото; после успеха удаляем сообщение с фото (best-effort) и в панели ✅ статус.

User photo хранится и реюзится (active), можно удалить/заменить/добавить.

Карта экранов (панелей)
0) HOME / Dashboard (главная)

Задача: всё видно с одного взгляда + быстрые переходы.

Текст панели (пример):

🧪 TryOnLab • Dashboard

🧠 Provider: Grok ✅ (video ✅)   |   Prompt: v1.0
👤 User photo: ✅ active (#2)   | stored: 3
🧥 Product photo: ✅ 1/1        | session id: 01J...
🎯 Fit: regular ✅
📏 Measurements: — (optional)
🧾 Last job: done ✅  (image)  | last video: —  

Actions:
• Generate = available ✅

Кнопки (inline keyboard):

Row1: 🧥 Товар | 👤 Моё фото

Row2: 🎯 Посадка | 📏 Параметры

Row3: ⚡ Генерировать

Row4: 🧾 История | ⚙️ Настройки | 🧼 Сброс сессии

Логика доступности:

⚡ Генерировать disabled, если нет product photo или нет active user photo.

В шапке provider: показывать video ✅/❌ по capability провайдера (у Grok video есть; у OpenAI зависит от конфигурации) 

PR-06

1) PRODUCT (загрузка фото товара)

Принцип: нажал “🧥 Товар” → панель переключилась в режим ожидания фото. Никаких “нажмите загрузить”.

Текст панели:

🧥 Товар • Загрузка

Пришли 1 фото товара (front) в этот чат.
После успешной загрузки я удалю сообщение с фото, а в панели появится ✅.

Текущий статус: ✅ 1/1 (есть)

Кнопки:

Row1: 🗑️ Очистить товар | 🔁 Заменить

Row2: ⬅️ Назад

Поведение:

Любое фото от юзера в этом состоянии = попытка сохранить как product.

Успех → save to storage (session scope) → delete message (best-effort) → panel update: ✅ 1/1.

Ошибка (не фото / > MAX_MB) → панель показывает ❌ и краткую причину.

V2: расширение до 3 слотов Front/Side/Back, но в UI можно сразу заложить “1/3”, пока скрыв остальные.

4) MEASUREMENTS (рост/мерки)

Ты хотел “нажал — сразу инпут”. В Telegram “авто-фокус” не гарантирован, но работает нормально через ForceReply (или просто “вводи сейчас, я жду”).

Текст панели:

Введи строкой:
рост 182, грудь 106, талия 86, бедра 102, плечи 50, inseam 84

Или нажми “Пропустить”.

Кнопки:

Row1: ⛔ Пропустить | 🗑️ Очистить

Row2: ⬅️ Назад

Поведение:

Нажал сюда → bot ставит FSM await_text и (опционально) отправляет ForceReply “Введи параметры…”.

После парсинга → удалить сообщение пользователя с мерками не обязательно (это не фото), можно оставить; но если хочешь чистоту, тоже можно best-effort.

При выборе элемента:

панель показывает детали + действия:

🔁 Retry с теми же входами (если inputs сохранены/доступны)

🎬 Видео 1..5 (если это image и есть ключ)

Правила поведения (чтобы это было “как OwnerBot/SocialBridge”)
A) “Одна панель”

У пользователя есть panel_message_id.

Любой callback редактирует эту панель.

Никаких вторых панелей “на всякий случай”.

B) Ожидание ввода без лишних шагов

Нажал “Товар” → state await_product_photo и сразу “пришли фото”.

Нажал “Моё фото → загрузить новое” → state await_user_photo.

Нажал “Параметры” → state await_measurements_text (и ForceReply, если хочешь удобнее).

C) Удаление входных фото

После сохранения файла в storage: try_delete(message_id) (best-effort).

Результаты не удаляем.

D) Реюз user photo

Храним N фото, есть active.

Generate всегда берёт active, пока юзер не сменил.

Быстрая кнопка “заменить active”.

E) Ошибки должны быть “коротко и злые”

В панели, например:

❌ Слишком большой файл (15MB). Лимит: 12MB.

❌ Это не фото. Пришли JPG/PNG/WebP.

❌ Provider rate limit. Переключись на OpenAI или попробуй позже.

Что это даёт (и почему это правильно)

Чат чистый: входные фото исчезают, панель одна, результаты остаются “галереей”.

UX без лишних кнопок: нажал раздел → сразу действие (ввод/фото).

Меньше “дуракавалидации”: панель всегда показывает ✅/❌ что готово.

Реюз user photo = меньше трения и стабильнее результат.
