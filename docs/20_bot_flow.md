# 20_bot_flow — TryOnLab (Telegram UX / Wizard)

## 0) Принципы UX (как в OwnerBot/SocialBridge)
1) **Одна “панель” (panel message)** на пользователя, которая **редактируется**, а не плодится.
2) Все действия через **inline-кнопки** (callback query). Текстом пользователь вводит только то, что неизбежно: параметры тела.
3) **Шаги назад/вперёд**: единая FSM-логика, панель всегда показывает текущий шаг и статус.
4) **Антидубли**: любые повторные клики по кнопкам должны быть идемпотентны (ничего не ломают).
5) **Чистка чата**:
   - **фото товара** и **фото человека**, после успешного принятия системой, **пытаемся удалить из чата**
   - **результаты генерации** (image/video) **не удаляем** (пусть висят как “галерея” пользователя)

> Удаление пользовательских сообщений в Telegram зависит от контекста/разрешений. Мы делаем `try_delete(message_id)` и **молча игнорируем ошибки** (например, если удалить нельзя).

---

## 1) Роли сообщений
### 1.1 Panel message (главное)
- Один message на пользователя, хранится как `panel_message_id`.
- Всегда `edit_text + edit_reply_markup`.
- Содержит:
  - текущий шаг
  - краткий статус загруженных ассетов
  - выбранный провайдер
  - кнопки управления

### 1.2 Ephemeral uploads (временные)
- Сообщения пользователя с фото товара/человека.
- После успешного сохранения в storage — **удаляем** (если возможно).
- В панели рисуем: ✅ загружено + размер/кол-во фото.

### 1.3 Result messages (постоянные)
- Отправляются отдельными сообщениями (photo/video).
- Не удаляются ботом.
- В подписи (caption) минимум:
  - provider
  - fit
  - (опц.) мерки/рост
  - время генерации
  - кнопки: 🔁 retry, 🎬 presets 1..5, ⚙️ settings

## 2) Основной wizard (V1)

### 2.1 Entry points
- `/start` → создаём панель или переиспользуем существующую
- `/new` → сбросить текущую сессию (не трогаем сохранённые user-фото)
- `/settings` → настройки (provider, user photos, retention)

### 2.2 Шаги
**Step 0 — Home**
Панель показывает:
- Provider: Grok | OpenAI
- User photos: N (active set: 1)
- Garment photos: 0
- Fit: not set
- Measurements: not set

Кнопки:
- `➕ Товар (загрузить фото)`
- `👤 Моё фото (выбрать/загрузить)`
- `🎯 Посадка (slim/regular/oversize)`
- `📏 Параметры (рост/мерки)`
- `⚡ Генерировать` (disabled пока нет: garment + user)
- `⚙️ Провайдер`
- `🧼 Сбросить сессию`

---

**Step 1 — Upload garment photo(s)**
Пользователь отправляет 1 фото товара.
V1: принимаем **1 фото**. (В V2 расширим до front/side/back)

После получения:
- сохраняем в storage: `tryon/{session_id}/input/product.jpg`
- записываем в session: `product_media_key`
- пытаемся удалить исходное сообщение пользователя с фото
- обновляем панель: ✅ “Товар: загружен (1)”

Кнопки панели:
- `➕ Загрузить заново` (перезаписать продукт)
- `⬅️ Назад`

---

**Step 2 — User photo (reuse-first)**
Здесь нюанс: пользовательское фото мы **можем хранить и реюзать**.

Панель:
- “Выберите фото пользователя для примерки:”
- показывает, есть ли сохранённые фото (N)

Кнопки:
- `✅ Использовать сохранённое` (если N>0 → открывает список)
- `⬆️ Загрузить новое`
- `🗑️ Удалить сохранённые`
- `⬅️ Назад`

#### 2.2.1 Список сохранённых user-фото (если их несколько)
- Пагинация/список кнопками:
  - `Фото #1 (активное)`
  - `Фото #2`
  - `Фото #3`
  - `➕ Добавить новое`
  - `⬅️ Назад`

Выбор фото:
- ставим `user_photo_id` как active для текущей сессии
- панель обновляем: ✅ “Фото человека: выбрано (id=…)”

#### 2.2.2 Загрузка нового user-фото
Пользователь отправляет фото.
После успешного принятия:
- сохраняем в storage: `users/{tg_user_id}/photos/{photo_id}.jpg`
- добавляем запись в `user_photos` (DB)
- ставим active = новый photo_id
- пытаемся удалить исходное сообщение пользователя с фото
- панель: ✅ “Фото человека: загружено и сохранено (N)”

> Опционально: ограничить до `MAX_USER_PHOTOS` (например 5). При превышении: просим удалить старые.

---

**Step 3 — Fit preference**
Inline кнопки:
- `Slim`
- `Regular`
- `Oversize`
- `⬅️ Назад`

После выбора:
- сохраняем `fit_pref` в session/job draft
- панель: ✅ “Посадка: regular” (например)

---

**Step 4 — Measurements (опционально)**
Кнопки:
- `✍️ Ввести рост/мерки`
- `⛔ Пропустить`
- `⬅️ Назад`

Если “ввести”:
- ожидаем текст формата:
  - `рост 182, грудь 106, талия 86, бедра 102, плечи 50, inseam 84`
- парсим максимально терпимо:
  - допускаем разделители `, ; /`
  - допускаем “рост=182”
  - сохраняем как JSON
- панель: ✅ “Параметры: рост 182, (3 мерки)”

---

**Step 5 — Generate (Image)**
Кнопка `⚡ Генерировать` становится активной только если:
- есть `product_media_key`
- есть активное `user_photo_id` (или `user_media_key`)

При запуске:
- создаём job через API `/jobs` (type=tryon_image)
- панель: `⏳ Генерация изображения…` + индикатор (псевдо-прогресс)
- запускаем polling (bot → GET /jobs/{id}) или push через webhooks (позже)

По завершении:
- отправляем **result message** с картинкой (не удаляем)
- панель возвращаем в Home + добавляем кнопку:
  - `🎬 Видео (1..5)` (активна после image)
  - `🔁 Регенерировать image`

---

## 3) Ретраи и стабильность
### 3.1 Регенерация (Retry)
- Кнопка `🔁 Регенерировать` создаёт:
  - либо `POST /jobs/{job_id}/retry`
  - либо новый job `tryon_image` с теми же входами + новый seed
- Результат приходит как новое сообщение (история не теряется).

### 3.2 “Стабильность результата”
Чтобы пользователю не грузить фото каждый раз:
- по умолчанию используем **последнее активное user-фото**
- в Home показываем: “Фото человека: active #X”
- кнопка “👤 Моё фото” позволяет заменить/добавить/удалить

---

## 4) Видео режим (presets 1..5)
### 4.1 Доступность
Видео доступно если:
- есть хотя бы один успешный image-result (последний job_id)
- провайдер поддерживает видео (capability `video=True`)

### 4.2 Кнопки
В панели (или под result-image сообщением) выводим:
- `🎬 1` `🎬 2` `🎬 3` `🎬 4` `🎬 5`

Нажатие:
- создаёт video job:
  - `POST /jobs/{image_job_id}/video?preset=N`
- панель: `⏳ Генерация видео (preset N)…`

По готовности:
- отправляем видео отдельным сообщением (не удаляем)
- панель: Home

---

## 5) Настройки (inline)
### 5.1 Провайдер (toggle)
Кнопка `⚙️ Провайдер` открывает меню:
- `✅ Grok` / `OpenAI`
- `⬅️ Назад`

При выборе:
- сохраняем в DB `user_settings.provider`
- обновляем панель “Provider: …”
- текущая сессия начинает использовать выбранного провайдера

> Также есть `AI_PROVIDER_ALLOWLIST`. Если провайдер выключен конфигом, кнопка сереет/скрывается.

### 5.2 Управление user-фото
Кнопка `👤 Моё фото` открывает:
- список сохранённых
- `➕ добавить`
- `🗑️ удалить выбранное`
- `🧨 удалить все`
- `⬅️ назад`

Удаление:
- удаляем запись из DB + key из storage
- если удалили active — выбираем следующий или требуем загрузку

---

## 6) Сессия и хранение состояния
### 6.1 Session (оперативное)
Храним в Redis (или in-memory для локального теста):
- `panel_message_id`
- `step`
- `product_media_key`
- `active_user_photo_id`
- `fit_pref`
- `measurements_json`
- `last_image_job_id`
- `last_result_image_url`

TTL сессии: например 24 часа.
Сохранённые user-фото не зависят от TTL.

### 6.2 User photos (долговременное)
Таблица `user_photos` (V1 минимально):
- `id`
- `user_id`
- `storage_key`
- `created_at`
- `is_active` (или активное хранить в `user_settings`)

---

## 7) “Панель” паттерн: технические требования
### 7.1 Функции-утилиты (обязательные)
- `ensure_panel(user_id) -> panel_message_id`
  - если панели нет: sendMessage + сохранить id
  - если есть: просто вернуть id
- `render_panel(state) -> (text, keyboard)`
- `safe_edit(panel_id, text, keyboard)`
  - ловит `MessageNotModified` и молча игнорирует
- `try_delete(chat_id, message_id)`
  - ловит `TelegramBadRequest` и игнорирует
- `answer_callback(query_id)` всегда (чтобы Telegram не “крутил”)

### 7.2 Callback data
Строго кодируем:
- `nav:home`
- `nav:garment`
- `nav:userphoto`
- `fit:slim|regular|oversize`
- `provider:grok|openai`
- `gen:image`
- `gen:retry`
- `video:1..5`
- `userphoto:select:{id}`
- `userphoto:delete:{id}`
- `session:reset`

> Все callback handlers должны быть идемпотентными: повторный клик не создаёт хаос.

---

## 8) Политика удаления сообщений (требование пользователя)
- **Входные фото** (товар/человек):  
  удаляем после успешной сохранности в storage и отметки ✅ в панели.
- **Результаты** (изображения/видео):  
  **не удаляем** никогда. Пользователь сам решает, чистить ему чат или жить с этим.

---

## 9) Privacy / управление данными пользователя
### 9.1 Реюз user-фото
По умолчанию **сохраняем** пользовательское фото для повторного использования:
- экономит время
- повышает стабильность результата (одна и та же база)

Но обязательно:
- меню “Удалить” (одно или все)
- понятная подпись “Фото сохранено, можно удалить в настройках”
- политика TTL/retention описана в `70_security_privacy.md`

### 9.2 Retention
V1 рекомендация:
- user-фото: хранить до удаления пользователем (или установить retention по умолчанию 30/90 дней)
- job media: хранить по `RETENTION_HOURS` (72 часа) для теста

---

## 10) Граничные случаи (обязательная обработка)
- пользователь нажал “Генерировать”, но нет user-фото → панель подсветить и вернуть на нужный шаг
- пользователь загрузил “не фото” → вежливо отказать и повторить step
- провайдер недоступен/лимит → job failed с retryable=true, в панели “попробуйте позже / смените провайдер”
- пользователь спамит кнопку “видео” → rate-limit + мягкое сообщение в панели

---

## 11) Итог: Definition of Done для bot flow V1
- Один panel message, всё через inline, без дублей.
- Загрузка product photo → удаление сообщения → отметка ✅ в панели.
- User photo: можно загрузить и сохранить, можно переиспользовать, можно удалить.
- Generate image → результат отдельным сообщением, панель возвращается в Home.
- Retry image → новый результат отдельным сообщением.
- Video presets 1..5 → видео отдельным сообщением.
- Provider toggle Grok/OpenAI работает и хранится в DB.
